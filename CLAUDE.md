# Lifly 项目准则

---

## 一、核心原则

1. **专业标准** — 做任何工作前，先问自己三个问题：①我现在是什么角色？②这个角色的专业水准是什么样的？③在专业水准下，这件事应该怎么做？按专业水准执行，而不是"够用就行"。

2. **复盘机制** — 研发过程中所有不符合预期的情况（bug、测试误判、环境问题等），在解决后必须复盘根因，将教训整理为简洁的通用准则补充到本文件中。个案经验要转化为系统性防线。

3. **文档驱动** — `docs/prd.md`（产品需求）和 `docs/tdd.md`（技术设计）是项目唯一权威来源。所有变更必须同步更新到对应文档，先更新文档再提交代码。

---

## 二、开发规范

### 前后端一致性

- 修改 API 接口时，必须同步检查并更新**前端类型定义、调用方、后端处理方**三方的字段名和数据结构。以后端 struct 为权威来源。
- API 响应结构必须明确：后端返回数组就用数组接收，不假设分页格式。新增 API 调用时先验证实际响应结构。
- 查询参数名必须对齐后端：前端 params key 与后端 Query struct 字段名完全一致。
- 前端不假设后端未返回的字段存在：TypeScript interface 的每个字段都必须有后端对应。

### 代码规范

- 后端: Rust + Axum，遵循现有模块结构（identity / capability / tool / data / intelligence / common）
- 前端: React 19 + TypeScript + Vite，使用 pnpm
- 文档统一放在 `docs/superpowers/` 下（specs/ 和 plans/）

### Git

- 提交信息使用英文，格式: `type: description`（feat / fix / chore / docs / test）
- 不要在没有明确指示的情况下自动 push

## 三、测试规范

### 执行方式

- **所有测试必须由 AI 自主完成，禁止要求用户手动运行。** 包括单元测试、集成测试、E2E 测试等。
- 后端测试: `cd server && cargo test -- --test-threads=1`
- 前端测试: `cd web && pnpm test`
- E2E 测试: `cd web && npx playwright test --project=chromium`
- LLM 集成测试: `cd server && LLM_API_KEY=$LLM_API_KEY cargo test --test llm_integration -- --test-threads=1`

### 质量标准

- 测试的目的是**验证功能正确**，不是凑数量。每个测试必须覆盖完整的用户操作闭环（输入→处理→结果展示）。
- E2E 断言必须区分成功和失败状态。不能用宽泛选择器（如 `.alert`）同时匹配成功和错误。
- 前端代码写完后必须通过 E2E 验证与后端的真实交互。类型定义、参数名、响应结构都必须对照后端实际行为确认。
- 测试通过不等于功能正确——必须审查截图或输出，确认验证的是预期行为而非巧合匹配。
- **构造测试数据时，必须使用尽量真实的数据。** 不能用 1x1 像素图片测试 OCR，不能用 "test" 测试 NLP 解析。测试数据的质量直接决定测试的有效性。
- **涉及外部服务（LLM、支付等）的功能，必须至少做一次真实 API 调用验证。** 不能依赖 `on_failure=skip` 掩盖的"假通过"。种子数据中的模型名、API 地址等必须对照真实 API 确认可用。
- **E2E 必须验证业务产出物，而非仅验证流程状态。** 例如 OCR 测试必须断言提取出了具体字段值，不能只检查 pipeline 状态为 completed。
- **涉及异步消息（WebSocket、事件）的功能，必须按业务 ID 过滤。** 广播事件在并发场景下会导致串扰。
- **修复错误时，必须先精确定位差异（对比期望值和实际值），再做最小修改。** 不能跳过分析直接替换为"能用的替代方案"。信息就在眼前时（如 API 返回的错误信息、文档中的命名规则），必须认真阅读并据此修正，而非降级绕过。

## 四、环境配置

- 端口 8080 是 coder 进程占用的，**绝对不能 kill**，否则整个会话崩溃
- Docker 服务映射到端口 9527（9527:8080）
- 数据库: PostgreSQL (pgvector/pgvector:pg16)
