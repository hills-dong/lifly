# Lifly — 产品需求文档 (PRD)

> Version: 0.3 | Date: 2026-03-09 | Status: Draft
>
> 读者：AI 辅助开发。本文档作为项目的权威产品定义，指导后续架构设计与工程实施。

---

## 1. 项目概述

**Lifly**（Life Fly）是一套自托管的智能个人生活管理系统。

用户在自建服务器上部署 Lifly，通过多端（Web / 桌面 / 手机 / 浏览器插件）采集生活中的碎片化数据，系统经 LLM 处理后结构化存储，并提供浏览、搜索、提醒、主动建议等消费能力。

**核心假设**：用户的生活管理需求是无限的、不可预见的。系统不应为每个场景预设固定功能，而应提供一套可动态组合的原子能力框架，由 LLM 根据用户意图即时组装为可用的工具。

**目标用户**：具备基本技术能力（能部署 VPS 或 NAS）、重视数据隐私、有多维度生活管理需求的个人用户或家庭。

---

## 2. 术语定义

| 术语 | 定义 |
|------|------|
| **原子能力 (Atomic Capability)** | 系统中最小的、不可再拆分的功能单元。不含业务逻辑，仅完成单一技术动作（如 OCR、语音转文本、写入数据库）。按职能分为采集、处理、存储、使用四类 |
| **工具 (Tool)** | 由一个或多个原子能力按特定顺序编排而成的功能单元，面向具体业务场景。如"Todo List"、"照片管理"、"证件管理"均为工具 |
| **数据管道 (Pipeline)** | 数据从采集到消费的完整处理流程。遵循 Collect → Process → Store → Use 四阶段。每次管道执行对应一个 Pipeline 实例 |
| **数据对象 (DataObject)** | 工具产出的业务数据单元。所有工具共用同一种数据容器，通过动态属性（JSON）区分不同场景的数据结构 |
| **原始输入 (RawInput)** | 客户端采集并提交到服务端的未处理数据。经 Pipeline 处理后转化为 DataObject |
| **用户记忆 (UserMemory)** | 系统通过长期交互积累的用户画像、偏好和习惯。存储在本地，用于个性化建议 |
| **演化引擎 (Evolution Engine)** | 基于用户反馈和使用数据，自动优化已有工具或生成新工具的系统机制 |
| **模板市场 (Template Market)** | 去隐私化后的工具模板集合，支持跨用户共享工具逻辑 |

## 3. 待解决的问题

个人生活中产生大量碎片化数据。现有工具存在三个核心问题：

1. **工具碎片化** — 照片、记账、笔记、健康等数据散落在十几个 App 中，无法关联分析
2. **功能僵化** — 每个工具只做预设功能，用户有新需求时只能换工具或妥协
3. **数据失控** — 个人隐私数据存储在各家云端，用户无法掌控

---

## 4. 核心理念

### 4.1 数据主权

所有数据和用户记忆存储在用户自建服务器。初期推理能力通过远程 LLM API 获取，但仅发送脱敏后的处理请求，数据本身不离开本地。随端侧模型成熟逐步迁移至完全本地推理。

### 4.2 意图驱动

系统不预设固定工具。用户用自然语言描述需求，LLM 理解意图后从原子能力库中挑选、组合、验证，动态生成可用工具。

### 4.3 能力演化

工具不是终态。系统通过用户反馈持续优化工具，通过类比推理在相似场景间复用逻辑，通过去隐私化实现跨用户的能力共享。

## 5. 系统架构概览

### 5.1 部署拓扑

```
                    ┌────────────────────────────┐
                    │     Self-Host Server        │
                    │                             │
                    │  ┌────────┐  ┌───────────┐  │
                    │  │  Data  │  │ Capability │  │
                    │  │ Store  │  │  Engine    │  │
                    │  └────────┘  └─────┬─────┘  │
                    │                    │        │
                    │          ┌─────────┼──────┐ │
                    │          ▼         ▼      ▼ │
                    │       Local    Remote  Remote│
                    │       LLM     API #1  API #2│
                    │      (Future)               │
                    └───┬─────┬──────┬──────┬─────┘
                        │     │      │      │
                      Web  Desktop Mobile Plugins
                      (RW)  (RW)    (W)    (W)
```

- **Self-Host Server**：核心节点。持久化存储、LLM 调度、能力编排
- **Web Client**：数据采集 + 数据管理（读写）
- **Desktop Client**：本地数据采集 + 管理（读写）
- **Mobile Client**：移动端数据采集（只写）
- **Software Plugins**：浏览器/系统事件采集（只写）

### 5.2 数据管道

所有数据遵循统一的四阶段管道：

```
Collect → Process → Store → Use
```

| 阶段 | 执行方 | 说明 |
|------|--------|------|
| **Collect** | 客户端 / 插件 | 采集多模态原始数据（图片、音频、文本、传感器、URL 等），提交 RawInput |
| **Process** | Server | 由 Pipeline 驱动原子能力编排处理。支持 LLM 自动处理 + Human-in-the-loop 人工审核。入库的是结构化数据，而非原始数据 |
| **Store** | Server | 文件系统（二进制文件）+ 关系数据库（结构化元数据）+ 向量库（语义嵌入） |
| **Use** | 用户 | 浏览、语义搜索、提醒、分享、LLM 主动建议 |

## 6. 能力演化引擎

### 6.1 概念模型

```
用户意图 ──→ 认知中枢 (The Brain)
              │  意图解析 → 需求澄清 → 工具架构设计
              │  ← 用户记忆/偏好辅助
              ▼
            能力工厂 (The Factory)
              │  原子能力编排 → 沙箱验证 → 合规检查 → 发布
              │
              ├──→ 原子能力库 (The DNA)
              │    采集 · 处理 · 存储 · 使用
              │
              └──→ 能力图谱 (The Ecosystem)
                   已生成工具 · 模板市场 · 演化谱系
```

### 6.2 原子能力分类

| 类别 | 职责 | 示例 |
|------|------|------|
| 采集 | 从外部获取原始数据 | OCR、语音流监听、摄像头帧截取、传感器读取、API 拉取 |
| 处理 | 对数据进行理解和转换 | 实体抽取、分类、翻译、摘要、情感分析、多模态对齐 |
| 存储 | 将处理结果持久化 | 关系型写入、向量化入库、时序存储、加密文件存储 |
| 使用 | 将数据呈现给用户或触发动作 | 动态 UI 生成、报表输出、推送通知、自动化控制 |

### 6.3 工具生命周期

```
描述需求 → LLM 澄清 → 架构设计 → 原子编排 → 沙箱验证 → 发布
    ↑                                                    │
    └── 用户反馈 → 演化引擎 → 新版本 → 验证 → 发布 ──────┘
                                                         │
                                          去隐私化 → 模板市场
```

### 6.4 演化引擎成熟度

> 以下是演化引擎自身的能力阶段，非产品整体 Roadmap。

| 阶段 | 定位 | 能力 |
|------|------|------|
| **V1 工具** | LLM 作为配置员 | 用户描述需求，LLM 挑选并串联原子能力，生成可用工具 |
| **V2 助手** | 具备长短期记忆 | 根据用户习惯和历史反馈，主动建议生成或优化工具 |
| **V3 生命体** | 能力图谱规模化 | 跨用户共享逻辑（无隐私数据），类比推理自动覆盖长尾场景 |

## 7. 场景模式与示例

### 7.1 通用模式

所有场景本质上都遵循同一种模式：

```
多模态输入 → LLM 结构化理解 → DataObject 入库 → 智能消费
```

不同场景的差异仅在于：

| 差异维度 | 举例 |
|----------|------|
| 输入类型 | 图片（照片/证件/小票）、音频（想法/梦境）、文本（链接/代码）、传感器数据 |
| 处理原子组合 | OCR + 分类、语音转写 + 实体抽取、图像识别 + 营养计算 |
| DataObject 属性结构 | 照片有 GPS/EXIF，证件有证件号/有效期，联系人有生日/偏好 |
| 消费方式 | 语义搜索、到期提醒、穿搭推荐、财务报表、降价告警 |

### 7.2 代表性场景

> 以下场景仅为示例。实际场景由用户意图动态生成，不限于此。

| 场景 | 输入 | 处理 | 消费 |
|------|------|------|------|
| Todo List | 文本输入 | 结构化解析 | 查看、提醒、完成追踪 |
| 照片管理 | 手机拍照 | 去重、标注、分类 | 语义搜索、高光推荐 |
| 证件管理 | 拍照证件 | 裁剪、OCR | 查看、复制号码、导出 PDF |
| 想法管理 | 手表录音 | 语音转文本 | 查看、转 Todo/项目 |
| 订阅管理 | 浏览器插件采集 | 提取周期/价格 | 花费汇总、到期提醒 |
| 财务票据 | 扫描小票 | 提取金额/分类 | 报表、报销打包 |
| 健康饮食 | 拍餐前照 | 食材识别、营养估算 | 营养提醒、周报 |
| 人际关系 | 语音录入 | 提取人物/事件 | 生日提醒、个性化建议 |
| 购物决策 | 保存链接 | 评价汇总、价格追踪 | 降价提醒 |

### 7.3 端到端示例

> 用户："帮我管管我女儿的阅读情况。"

1. **澄清** — "您是想记录书单，还是每天读了多少页？需要生成读后感小册子吗？"
2. **生成** — 编排 [书脊 OCR] + [阅读计时器] + [PDF 生成] + [推送通知]，生成"少儿阅读追踪器"工具
3. **使用** — 拍照扫书自动获取简介，每晚提醒打卡
4. **演化** — 发现用户关注视力保护，自动加入 [环境光线监测]，光线不足时告警
5. **扩散** — 去隐私化后推荐给其他家长用户

## 8. 产品 Roadmap

| 里程碑 | 目标 | Scope | 验证点 |
|--------|------|-------|--------|
| **M1 骨架验证** | 管道跑通 + 架构可扩展 | Server + REST API + Web Client + Todo List 场景（预置工具）。架构设计需保证后续新增场景不改代码和数据库 | Todo 端到端完成；手动注册第二个场景时零代码变更 |
| **M2 多端 + 能力丰富** | 扩展数据入口和原子能力 | Mobile 采集端 + Browser Plugin + 更多预置场景 + 原子能力积累 | 多设备采集汇入管道；原子能力库具备一定覆盖面 |
| **M3 LLM 接入** | 验证意图驱动 | 远程 LLM 接入 + 从自然语言动态生成工具 | 用户用自然语言描述需求，系统自动组装出可用工具 |
| **M4 智能闭环** | 验证记忆与演化 | 用户记忆 + 主动建议 + 反馈驱动工具迭代 | 系统基于历史数据给出主动建议 |

## 9. 约束与边界

### 9.1 技术约束

| 约束 | 说明 |
|------|------|
| 一人开发 | 技术选型必须轻量、成熟、社区活跃，避免同时维护多套技术栈 |
| 后端为主 | 前端能力有限，Web Client 优先选择低复杂度方案 |
| 远程 LLM 依赖 | 初期推理能力依赖远程模型 API（如 OpenAI、Claude），需处理网络延迟和 API 成本 |
| 自托管部署 | 必须支持 Docker 一键部署，不依赖特定云平台服务 |

### 9.2 Scope 排除项（M1 明确不做）

- 不做 Desktop Client
- 不做 Mobile Client
- 不做 Browser Plugin
- 不做 LLM 动态工具生成
- 不做模板市场
- 不做跨用户共享
- 不做用户记忆和主动建议

### 9.3 依赖假设

| 假设 | 风险 |
|------|------|
| 用户具备部署 Docker 的能力 | 可能限制用户群体 |
| 远程 LLM API 可用且稳定 | API 价格变动或服务中断的风险 |
| JSONB + 动态 schema 能满足性能需求 | 数据量大时 JSONB 查询性能待验证 |

## 10. 非功能需求

| 维度 | 要求 |
|------|------|
| 隐私 | 数据与用户记忆仅存本地；远程 LLM 仅接收脱敏请求 |
| 部署 | Docker 一键部署；支持 VPS / NAS |
| 扩展 | 新场景通过注册工具上线，不需要改已有代码和数据库 |
| 性能 | 数据管道异步处理，不阻塞客户端采集 |
| 安全 | 端到端加密传输；敏感文件加密存储；设备令牌认证 |
